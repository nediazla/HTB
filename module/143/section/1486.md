## Abusando de las ACL

Una vez más, para recapitular dónde estamos y dónde queremos llegar. Estamos en control del`wley`Usuario cuyo hash NTLMV2 recuperamos al ejecutar respondedor antes en la evaluación. Por suerte para nosotros, este usuario estaba usando una contraseña débil, y pudimos descifrar el hash fuera de línea usando hashcat y recuperar el valor de ClearText. Sabemos que podemos usar este acceso para iniciar una cadena de ataque que nos dará como resultado el control del`adunn`El usuario que puede realizar el ataque DCSYNC, lo que nos daría el control total del dominio al permitirnos recuperar los hash de contraseña NTLM para todos los usuarios en el dominio y aumentar los privilegios al administrador de dominio/empresa e incluso lograr la persistencia. Para realizar la cadena de ataque, tenemos que hacer lo siguiente:

1. Usar el`wley`usuario para cambiar la contraseña para el`damundsen`usuario
2. Autenticarse como el`damundsen`usuario y apalancamiento`GenericWrite`derechos para agregar un usuario que controlamos al`Help Desk Level 1`grupo
3. Aprovechar la membresía del grupo anidada en el`Information Technology`grupo y apalancamiento`GenericAll`derechos para tomar el control del`adunn`usuario

Entonces, primero, debemos autenticarnos como`wley`y forzar cambiar la contraseña del usuario`damundsen`. Podemos comenzar abriendo una consola de PowerShell y autenticando como el`wley`usuario. De lo contrario, podríamos omitir este paso si ya estábamos ejecutando como este usuario. Para hacer esto, podemos crear un[Objeto PSCredential](https://docs.microsoft.com/en-us/dotnet/api/system.management.automation.pscredential?view=powershellsdk-7.0.0).

#### Creación de un objeto PSCredential

  Tácticas de abuso de ACL

```powershell-session
PS C:\htb> $SecPassword = ConvertTo-SecureString '<PASSWORD HERE>' -AsPlainText -Force
PS C:\htb> $Cred = New-Object System.Management.Automation.PSCredential('INLANEFREIGHT\wley', $SecPassword) 
```

A continuación, debemos crear un[Objeto Securestring](https://docs.microsoft.com/en-us/dotnet/api/system.security.securestring?view=net-6.0)que representa la contraseña que queremos establecer para el usuario de destino`damundsen`.

#### Creación de un objeto Securestring

  Tácticas de abuso de ACL

```powershell-session
PS C:\htb> $damundsenPassword = ConvertTo-SecureString 'Pwn3d_by_ACLs!' -AsPlainText -Force
```

Finalmente, usaremos el[Set DomainuserPassword](https://powersploit.readthedocs.io/en/latest/Recon/Set-DomainUserPassword/)Función PowerView para cambiar la contraseña del usuario. Necesitamos usar el`-Credential`bandera con el objeto de credencial que creamos para el`wley`usuario. Es mejor especificar siempre el`-Verbose`Marcar para obtener comentarios sobre el comando que complete como se esperaba o tanta información sobre los errores como sea posible. Podríamos hacer esto desde un host de ataque de Linux usando una herramienta como`pth-net`, que es parte del[Pth-Toolkit](https://github.com/byt3bl33d3r/pth-toolkit).

#### Changing the User's Password

  Tácticas de abuso de ACL

```powershell-session
PS C:\htb> cd C:\Tools\
PS C:\htb> Import-Module .\PowerView.ps1
PS C:\htb> Set-DomainUserPassword -Identity damundsen -AccountPassword $damundsenPassword -Credential $Cred -Verbose

VERBOSE: [Get-PrincipalContext] Using alternate credentials
VERBOSE: [Set-DomainUserPassword] Attempting to set the password for user 'damundsen'
VERBOSE: [Set-DomainUserPassword] Password for user 'damundsen' successfully reset
```

Podemos ver que el comando completado correctamente, cambiando la contraseña para el usuario de destino mientras usa las credenciales que especificamos para el`wley`usuario que controlamos. A continuación, necesitamos realizar un proceso similar para autenticar como el`damundsen`usuario y agréguenos al`Help Desk Level 1`grupo.

## Limpieza

En términos de limpieza, hay algunas cosas que debemos hacer:

1. Eliminar el SPN falso que creamos en el`adunn`usuario.
2. Eliminar el`damundsen`usuario del`Help Desk Level 1`grupo
3. Establezca la contraseña para el`damundsen`El usuario vuelve a su valor original (si lo conocemos) o que nuestro cliente lo establezca/alerta al usuario

Este pedido es importante porque si eliminamos primero al usuario del grupo, entonces no tendremos los derechos para eliminar el SPN falso.

Primero, eliminemos el SPN falso del`adunn`cuenta.

A pesar de que realizamos la mayor cantidad de limpieza posible, deberíamos incluir todas las modificaciones que realizamos en nuestro informe de evaluación final. Nuestro cliente querrá ser informado de cualquier cambio dentro del entorno, y la grabación de todo lo que hacemos durante una evaluación por escrito ayuda a nuestro cliente y a nosotros las preguntas para que surjan.

Este es solo un camino de ataque de ejemplo. Podría haber muchos caminos de ataque en un dominio grande, algunos más cortos y algunos más complicados. Si bien este camino era ficticio para este entorno de laboratorio específico, he visto caminos de ataque similares durante los compromisos del mundo real, y los ataques de LCA a menudo entran en juego para promover el acceso. Sin embargo, a veces, una cadena de ataque de ACL puede ser demasiado lento o potencialmente destructiva, por lo que podemos preferir enumerar el camino para presentar a nuestro cliente con suficiente evidencia para comprender el problema y realizar la remediación.

## Detección y remediación

Algunas recomendaciones sobre las ACL incluyen:

1. `Auditing for and removing dangerous ACLs`

Las organizaciones deben tener auditorías publicitarias regulares, pero también capacitar al personal interno para ejecutar herramientas como Bloodhound e identificar ACL potencialmente peligrosas que se pueden eliminar.

2. `Monitor group membership`

La visibilidad en grupos importantes es primordial. Todos los grupos de alto impacto en el dominio deben ser monitoreados para alertar al personal de los cambios que podrían ser indicativos de una cadena de ataque de ACL.

3. `Audit and monitor for ACL changes`

Habilitando el[Política de auditoría de seguridad avanzada](https://docs.microsoft.com/en-us/archive/blogs/canitpro/step-by-step-enabling-advanced-security-audit-policy-via-ds-access)puede ayudar a detectar cambios no deseados, especialmente[ID de evento 5136: se modificó un objeto de servicio de directorio](https://docs.microsoft.com/en-us/windows/security/threat-protection/auditing/event-5136)lo que indicaría que el objeto de dominio se modificó, lo que podría ser indicativo de un ataque de ACL. Si observamos el registro de eventos después de modificar el ACL del objeto de dominio, veremos alguna ID de evento`5136`creado:
