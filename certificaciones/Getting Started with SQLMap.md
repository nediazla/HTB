# **Escenario básico**

En un escenario simple, un probador de penetración accede a la página web que acepta la entrada del usuario a través de un`GET`parámetro (por ejemplo,`id`). Luego quieren probar si la página web se ve afectada por la vulnerabilidad de inyección SQL. Si es así, desearían explotarlo, recuperar la mayor cantidad de información posible de la base de datos de back-end o incluso intentar acceder al sistema de archivos subyacente y ejecutar los comandos del sistema operativo. Un ejemplo de código PHP vulnerable SQLI para este escenario se vería de la siguiente manera:

Código:php

```php
$link = mysqli_connect($host, $username, $password, $database, 3306);
$sql = "SELECT * FROM users WHERE id = " . $_GET["id"] . " LIMIT 0, 1";
$result = mysqli_query($link, $sql);
if (!$result)
    die("<b>SQL error:</b> ". mysqli_error($link) . "<br>\n");

```

Como el informe de error está habilitado para la consulta SQL vulnerable, habrá un error de base de datos devuelto como parte de la respuesta del servidor web en caso de cualquier problema de ejecución de consulta SQL. Dichos casos alivian el proceso de detección de SQLI, especialmente en el caso de manipulación del valor del parámetro manual, ya que los errores resultantes se reconocen fácilmente:

![](https://academy.hackthebox.com/storage/modules/58/rOrm8tC.png)
